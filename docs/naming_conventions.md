# Naming Conventions

## Overview

This document defines the naming conventions used throughout the Data Warehouse project.  
It describes the rules applied when naming schemas, tables, views, columns, surrogate keys, technical columns, and stored procedures.

The purpose of these conventions is to ensure consistency, readability, maintainability, and scalability across all layers of the Data Warehouse.

# General Principles

## Naming Style

- Use **snake_case** naming convention.
- Use lowercase letters for all object names.
- Use underscores (`_`) to separate words.

Example:

```
customer_information
sales_amount
product_key
```

---

## Language

- All database objects must use **English names**.
- Avoid using Arabic or mixed-language names.

---

## Reserved Words

- Avoid using SQL reserved keywords as object names.

Examples of invalid names:

```
order
user
date
group
```

Use meaningful alternatives:

```
order_number
user_account
created_date
```

# Schema Naming Conventions

Schemas should represent the Data Warehouse layer.

Pattern:

```
<layer_name>
```

Where:

- `<layer_name>` represents the processing stage of the data.

Examples:

```
bronze
silver
gold_layer
```

# Table Naming Conventions

## Bronze Layer Rules

The Bronze Layer stores raw data extracted from source systems.

### Naming Pattern

```
<sourcesystem>_<entity>
```

Where:

- `<sourcesystem>`: The name of the source system.
- `<entity>`: The original table name from the source system.

### Rules

- Keep the original source table names.
- Do not rename business entities.
- Add only the source system prefix.

### Examples

```
crm_cust_info
crm_prd_info
erp_loc_a101
```

Meaning:

```
crm_cust_info
→ Customer information table extracted from CRM system.
```

---

# Silver Layer Rules

The Silver Layer contains cleaned and standardized data.

### Naming Pattern

```
<sourcesystem>_<entity>
```

Where:

- `<sourcesystem>`: Source system name.
- `<entity>`: Original source entity name.

### Rules

- Maintain the same naming structure as Bronze Layer.
- Apply data cleansing and standardization without changing the business entity name.

### Examples

```
crm_cust_info
crm_prd_info
erp_px_cat_g1v2
```

---

# Gold Layer Rules

The Gold Layer represents business-ready data designed for reporting and analytics.

### Naming Pattern

```
<category>_<entity>
```

Where:

- `<category>`: Defines the table type.
- `<entity>`: Business-related descriptive name.

---

## Gold Table Prefixes

| Pattern | Meaning | Example |
|---------|---------|---------|
| dim_ | Dimension Table | dim_customers |
| fact_ | Fact Table | fact_sales |
| report_ | Reporting Table | report_monthly_sales |

---

## Examples

```
gold_layer.dim_customers
```

Represents:

```
Dimension table containing customer information.
```

---

```
gold_layer.dim_products
```

Represents:

```
Dimension table containing product information.
```

---

```
gold_layer.fact_sales
```

Represents:

```
Fact table containing sales transactions and measures.
```

# Column Naming Conventions

## General Rules

- Column names must use lowercase letters.
- Use snake_case format.
- Names should be meaningful and business-friendly.
- Avoid abbreviations unless they are commonly understood.

Examples:

Correct:

```
customer_number
sales_amount
shipping_date
```

Incorrect:

```
cust_no
salesAmt
ShipDate
```

---

# Surrogate Keys

All dimension table primary keys must use the suffix:

```
<table_name>_key
```

Where:

- `<table_name>` represents the entity name.
- `_key` indicates that the column is a surrogate key.

---

## Rules

- Surrogate keys are generated inside the Data Warehouse.
- They do not contain business meaning.
- They are used for relationships between dimensions and fact tables.

---

## Examples

Customer Dimension:

```
customer_key
```

Product Dimension:

```
product_key
```

Fact Table Foreign Keys:

```
customer_key
product_key
```

---

# Business Keys

Business keys represent the original identifiers from source systems.

Examples:

```
customer_id
product_id
customer_number
product_number
```

Rules:

- Preserve the original business identifier.
- Do not use business keys as primary keys in dimension tables.

# Technical Columns

Technical columns store metadata generated by the Data Warehouse system.

### Naming Pattern

```
dwh_<column_name>
```

Where:

- `dwh` indicates a system-generated metadata column.
- `<column_name>` describes the purpose of the column.

---

## Examples

| Column Name | Description |
|-------------|-------------|
| dwh_load_date | Date when the record was loaded into the Data Warehouse. |
| dwh_created_date | Date when the record was created in the warehouse. |
| dwh_update_date | Date when the record was last updated. |

---

# Stored Procedures

Stored procedures responsible for loading data must follow the naming pattern:

```
load_<layer>
```

Where:

- `<layer>` represents the Data Warehouse layer being loaded.

---

## Examples

```
load_bronze
```

Stored procedure responsible for loading data into the Bronze Layer.

```
load_silver
```

Stored procedure responsible for loading data into the Silver Layer.

```
load_gold
```

Stored procedure responsible for loading data into the Gold Layer.
